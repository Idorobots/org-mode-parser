Vantage is a *distributed in-memory cache* designed for sub-millisecond read
latency at high concurrency. It implements a superset of the <<<Redis>>> wire
protocol and exposes both a binary framing protocol and a human-readable
text interface.

Key properties of Vantage:

- *Consistent hashing* with virtual nodes for cluster topology.
- *Multi-master replication* with last-write-wins conflict resolution.
- *Pluggable eviction policies*: LRU, LFU, TTL-only, CLOCK.
- *Persistence* via append-only log with configurable fsync intervals.
- *TLS mutual authentication* on all inter-node and client connections.

** Vision and Goals
:PROPERTIES:
:CUSTOM_ID: vision
:END:

Vantage exists to fill the gap between Redis (single-master, complex cluster
sharding) and Memcached (no replication, limited data types). Our goals:

1. Drop-in Redis protocol compatibility for the most common commands.
2. Horizontal write scalability without a primary/replica split.
3. First-class Rust client with async/await support.
4. Observable by default: OpenTelemetry traces and Prometheus metrics out of the box.

** Non-Goals

We explicitly do *not* aim to:

- Replace Redis as a pub/sub broker or stream processor.
- Provide ACID transactional guarantees.
- Support arbitrary Lua scripting (=EVAL= is a non-goal for v1).
- Offer a managed cloud service (that is a downstream concern).

** System Components
:PROPERTIES:
:CUSTOM_ID: components
:END:

The Vantage system comprises five major components:

| Component         | Language | Role                                         |
|-------------------+----------+----------------------------------------------|
| ~vantage-server~  | Rust     | Core cache daemon, protocol handler          |
| ~vantage-cluster~ | Rust     | Cluster membership, gossip, topology         |
| ~vantage-client~  | Rust     | Async client library (Tokio)                 |
| ~vantage-cli~     | Rust     | Interactive REPL and scripting tool          |
| ~vantage-exporter~| Go       | Prometheus metrics bridge (sidecar)          |

*** vantage-server
:PROPERTIES:
:CUSTOM_ID: vantage-server
:END:

The server process owns the in-memory keyspace and handles all client
connections. It is structured as a tower of async middleware layers:

#+begin_example
TLS termination
    ↓
Protocol framing (Redis RESP3 / Vantage binary)
    ↓
Command dispatch
    ↓
ACL enforcement
    ↓
Keyspace mutation / read
    ↓
Replication fanout
    ↓
Response serialisation
#+end_example

*** vantage-cluster
:PROPERTIES:
:CUSTOM_ID: vantage-cluster
:END:

Cluster membership uses a variant of the SWIM[fn:swim] gossip protocol with
the following modifications:

- Piggyback propagation of keyspace metadata (not just membership events).
- Suspicion sub-protocol suppressed in favour of direct failure detection.
- Ring topology exposed as a =RwLock<Arc<Ring>>= for zero-copy reads.

[fn:swim] Das, A., Gupta, I., & Motivala, A. (2002). SWIM: Scalable weakly-consistent
infection-style process group membership protocol. /DSN 2002/.

*** vantage-client
:PROPERTIES:
:CUSTOM_ID: vantage-client
:END:

The async client uses connection pooling with configurable:

- Minimum idle connections per node.
- Maximum connections per node.
- Connection acquisition timeout.
- Health check interval.

#+begin_src rust
use vantage_client::{Client, Config};

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let config = Config::builder()
        .nodes(["cache-1:6380", "cache-2:6380", "cache-3:6380"])
        .tls_ca_cert("ca.crt")
        .pool_size(32)
        .build()?;

    let client = Client::connect(config).await?;
    client.set("hello", "world", None).await?;
    let val: Option<String> = client.get("hello").await?;
    println!("{val:?}");  // Some("world")
    Ok(())
}
#+end_src

** Data Flow
:PROPERTIES:
:CUSTOM_ID: data-flow
:END:

*** Write path

A write request follows this path through the system:

1. Client sends =SET key value [EX seconds]= in RESP3 format.
2. Server decodes the command and performs ACL check.
3. Consistent hash ring determines which /primary shard/ owns the key.
4. If the server is *not* the primary shard, the request is *proxied*.
5. Primary shard acquires a shard lock (=tokio::sync::RwLock=, write mode).
6. Value is written to the in-memory slab allocator.
7. Write is appended to the AOF (append-only file) if persistence is enabled.
8. Replication events are fanned out to replicas asynchronously.
9. =+OK= response is returned to the client.

*** Read path

1. Client sends =GET key=.
2. ACL check.
3. Consistent hash: locate primary shard.
4. Proxy if not local (or read from any replica if =read_replicas= is enabled).
5. Acquire shard lock (read mode).
6. Lookup in the slab, check TTL expiry.
7. Return value or =$-1= (nil bulk string).

** Consistent Hashing
:PROPERTIES:
:CUSTOM_ID: consistent-hashing
:END:

Vantage uses jump consistent hashing [cite:@lamping2014jump] with =V= virtual
nodes per physical node (default: =V = 150=).

The ring is rebalanced whenever a node joins or leaves. During rebalance:

- New key assignments are computed atomically.
- In-flight requests to old shard owners complete normally.
- Key migration runs in the background at a configurable rate.

** Replication Model
:PROPERTIES:
:CUSTOM_ID: replication
:END:

Each shard is replicated to =R= peers (default: =R = 2=) using asynchronous
multi-master replication. Conflict resolution is /last-write-wins/ based on
a Hybrid Logical Clock (HLC)[fn:hlc].

[fn:hlc] Kulkarni, S., Demirbas, M., et al. (2014). Logical physical clocks and
consistent snapshots in globally distributed databases. /OPODIS 2014/.

*** Replication lag monitoring

:NOTE:
Replication lag is exposed as a Prometheus gauge =vantage_replication_lag_seconds=.
Alert threshold: 500 ms sustained for > 60 s → PagerDuty P2.
:END:

** Eviction Policies
:PROPERTIES:
:CUSTOM_ID: eviction
:END:

| Policy       | Algorithm                    | Use case                         |
|--------------+------------------------------+----------------------------------|
| ~lru~        | Approximated LRU (samples=5) | General-purpose                  |
| ~lfu~        | Count-min sketch, decay=0.5  | Skewed access distributions      |
| ~ttl~        | Expires-bucket sorted set    | Time-bounded session caches      |
| ~clock~      | Two-handed CLOCK algorithm   | Low-overhead, uniform workloads  |
| ~noeviction~ | Return error when full       | Strict capacity guarantees       |
| ~allkeys-lru~| LRU across all keys          | When all keys may be evicted     |

** Persistence
:PROPERTIES:
:CUSTOM_ID: persistence
:END:

*** Append-Only File (AOF)

The AOF records every write command in RESP3 format, one command per line.
On restart, the server replays the AOF to reconstruct state.

AOF configuration options:

- ~aof_enabled~: =true= / =false= (default =false=).
- ~aof_path~: filesystem path (default =/var/lib/vantage/vantage.aof=).
- ~aof_fsync~: =always= / =everysec= / =no= (default =everysec=).
- ~aof_rewrite_min_size~: minimum AOF size before compaction (default =64MiB=).

*** Snapshots (RDB-style)

Point-in-time snapshots are written by forking the process and serialising
the keyspace to a compact binary format. Snapshot schedule is configured via
=snapshot_schedule= as a cron-like expression.

** Security Model
:PROPERTIES:
:CUSTOM_ID: security
:END:

*** TLS Configuration

All connections (client-to-server and inter-node) use TLS 1.3 with mandatory
mutual authentication. Certificates must be signed by the cluster CA.

: [tls]
: ca_cert = "/etc/vantage/ca.crt"
: server_cert = "/etc/vantage/server.crt"
: server_key = "/etc/vantage/server.key"
: require_client_cert = true
: min_version = "TLSv1.3"

*** Access Control Lists

ACL rules are evaluated in order; first match wins. Rule syntax:

: user alice on >password ~cache:* &* +@read +@write -DEBUG
: user readonly on >secret ~*  &* +@read
: user default off

** Functional Requirements
:PROPERTIES:
:CUSTOM_ID: functional-requirements
:END:

*** FR-01: Core Data Types

- [X] String (binary-safe, up to 512 MiB)
- [X] List (doubly linked list or compressed quicklist)
- [X] Hash (hash table with listpack encoding for small hashes)
- [X] Set (hash table with intset encoding for small integer sets)
- [X] Sorted Set (skip list + hash table, listpack for small zsets)
- [-] Stream (in progress — targeting v1.2)
- [ ] JSON (planned for v2.0 — requires RFC 8259 compliance)
- [ ] Bloom Filter (planned, via probabilistic module)

*** FR-02: Key Expiry

- [X] =EX seconds= — expire after N seconds.
- [X] =PX milliseconds= — expire after N milliseconds.
- [X] =EXAT unix-timestamp= — expire at Unix timestamp (seconds).
- [X] =PXAT unix-timestamp-ms= — expire at Unix timestamp (milliseconds).
- [X] =PERSIST= — remove expiry from a key.
- [X] =TTL key= — return remaining seconds.
- [X] =PTTL key= — return remaining milliseconds.

*** FR-03: Cluster Operations

- [X] Automatic cluster discovery via gossip.
- [X] Online node addition without downtime.
- [X] Online node removal with key migration.
- [-] Rolling cluster upgrade (partial — requires coordinator node).
- [ ] Cross-datacenter replication topology.

*** FR-04: Observability

- [X] =INFO= command (Redis-compatible output).
- [X] Prometheus metrics endpoint (=vantage-exporter= sidecar).
- [X] OpenTelemetry traces for all commands (opt-in, sampling configurable).
- [ ] Structured logging in JSON format (currently plain text).
- [ ] Integration with Grafana Cloud (dashboard JSON to be published).

*** FR-05: Client Protocol Compatibility

- [X] RESP2 (Redis Serialisation Protocol v2) — full compatibility.
- [X] RESP3 — full compatibility.
- [-] Inline commands — partial (single-line =GET/SET= only).
- [ ] Vantage binary protocol v1 (in design).

*** NFR-01: Latency
:PROPERTIES:

| Percentile | Target   | Measured (v0.9.4) |
|------------+----------+-------------------|
| p50        | < 0.1 ms | 0.08 ms           |

| p99.9      | < 5.0 ms | 3.21 ms           |

*** NFR-02: Throughput

Target: ≥ 500,000 operations/second per node on standard hardware
(AWS c6i.2xlarge or equivalent: 8 vCPU, 16 GiB RAM).

Current measured: 620,000 ops/sec (v0.9.4, read-heavy workload).

*** NFR-03: Memory Overhead

Memory overhead per key (excluding value size) must not exceed 128 bytes.
Current overhead: 84 bytes (slab metadata + hash table entry).

*** NFR-04: Availability

Target: 99.99% uptime (< 52.6 minutes downtime/year).
Achieved via: replica failover ≤ 5 seconds, health-check interval 1 second.

** String Commands
:PROPERTIES:
:CUSTOM_ID: string-commands
:END:

*** GET
:PROPERTIES:
:CUSTOM_ID: cmd-get
:END:

#+begin_example
GET key
#+end_example

Return the value associated with =key=, or nil if the key does not exist.
Time complexity: O(1). Returns an error if the value is not a string.

*** SET
:PROPERTIES:
:CUSTOM_ID: cmd-set
:END:

#+begin_example
SET key value [NX | XX] [GET] [EX seconds | PX ms | EXAT ts | PXAT ts | KEEPTTL]
#+end_example

Set =key= to =value=. Options:

- ~NX~ — only set if key does *not* exist.
- ~XX~ — only set if key *already* exists.
- ~GET~ — return the old value before overwriting.
- ~KEEPTTL~ — preserve existing TTL when overwriting.

Time complexity: O(1).

*** MGET / MSET
:PROPERTIES:
:CUSTOM_ID: cmd-mget-mset
:END:

#+begin_example
MGET key [key ...]
MSET key value [key value ...]
#+end_example

Atomic multi-key read and write. ~MSET~ is always successful. ~MSETNX~
(set multiple only if none exist) is also supported.
Time complexity: O(N) where N is the number of keys.

*** INCR / INCRBY / DECR / DECRBY
:PROPERTIES:
:CUSTOM_ID: cmd-incr
:END:

Increment or decrement integer-valued strings atomically.

#+begin_src
INCR counter         → integer reply
INCRBY counter 5     → integer reply
INCRBYFLOAT rate 0.1 → bulk string reply (float as string)
#+end_src

** List Commands
:PROPERTIES:
:CUSTOM_ID: list-commands
:END:

*** LPUSH / RPUSH
:PROPERTIES:
:CUSTOM_ID: cmd-lpush
:END:

#+begin_example
LPUSH key element [element ...]
RPUSH key element [element ...]
#+end_example

Push elements to the head (=L=) or tail (=R=) of a list.
Multiple elements are pushed in order: ~LPUSH mylist a b c~ results in =c b a= (c at head).

*** LRANGE
:PROPERTIES:
:CUSTOM_ID: cmd-lrange
:END:

#+begin_example
LRANGE key start stop
#+end_example

Return a range of elements. Indices are 0-based; negative indices count from the tail
(=-1= is the last element). O(S+N) where S is the offset from head/tail and N is range size.

*** LMOVE
:PROPERTIES:
:CUSTOM_ID: cmd-lmove
:END:

#+begin_example
LMOVE source destination LEFT|RIGHT LEFT|RIGHT
#+end_example

Atomically move an element from one list to another. Replaces the deprecated ~RPOPLPUSH~.

** Hash Commands
:PROPERTIES:
:CUSTOM_ID: hash-commands
:END:

*** HSET / HGET / HMGET
:PROPERTIES:
:CUSTOM_ID: cmd-hset
:END:

#+begin_example
HSET key field value [field value ...]
HGET key field
HMGET key field [field ...]
HGETALL key
#+end_example

Hash field operations. ~HSET~ now accepts multiple field-value pairs (since Redis 4.0).

*** HINCRBY / HINCRBYFLOAT

Increment hash field values atomically — identical semantics to ~INCR~/~INCRBYFLOAT~.

** Sorted Set Commands
:PROPERTIES:
:CUSTOM_ID: zset-commands
:END:

*** ZADD
:PROPERTIES:
:CUSTOM_ID: cmd-zadd
:END:

#+begin_example
ZADD key [NX|XX] [GT|LT] [CH] [INCR] score member [score member ...]
#+end_example

Add members with scores to a sorted set.

- ~GT~ — only update if new score is *greater* than current.
- ~LT~ — only update if new score is *less* than current.
- ~CH~ — return count of /changes/ (adds + updates) instead of only adds.

*** ZRANGEBYSCORE / ZRANGEBYLEX / ZRANGE
:PROPERTIES:
:CUSTOM_ID: cmd-zrange
:END:

~ZRANGE~ (added in Redis 6.2) unifies all range queries:

#+begin_example
ZRANGE key min max [BYSCORE|BYLEX] [REV] [LIMIT offset count] [WITHSCORES]
#+end_example

The older ~ZRANGEBYSCORE~ and ~ZRANGEBYLEX~ are deprecated aliases.

** Administrative Commands
:PROPERTIES:
:CUSTOM_ID: admin-commands
:END:

*** INFO

Returns server statistics in Redis-compatible format. Sections:

- =server= — version, uptime, OS, Rust version.
- =clients= — connected clients, blocked clients.
- =memory= — used memory, peak memory, allocator stats.
- =cluster= — cluster enabled, node count, ring version.
- =keyspace= — per-database key count and expiry statistics.

*** CLUSTER INFO / CLUSTER NODES

Returns cluster topology. =CLUSTER NODES= output is compatible with Redis
cluster tooling.

*** DEBUG SLEEP / DEBUG JMAP

:NOTE:
=DEBUG= commands are available only when the server is built with the
=debug-commands= feature flag. They must never be enabled in production.
:END:

:END:

1. Fork the repository and create a feature branch from =main=.
2. Ensure all tests pass: =cargo nextest run --all-features=.

** Architectural Decision Records
:PROPERTIES:
:CUSTOM_ID: adrs
:END:

ADRs document significant architectural decisions. They are stored in =docs/adr/=.

*** ADR-001: Adopt jump consistent hashing
:PROPERTIES:
:CUSTOM_ID: adr-001
:END:
CLOSED: [2024-03-15 Fri]

*Status:* Accepted.
*Decision:* Use jump consistent hashing over rendezvous hashing.
*Rationale:* Lower variance in key distribution; faster computation; well-studied
guarantees. Trade-off: less flexibility in heterogeneous node weights.

*** ADR-002: RESP3 as default client protocol
:PROPERTIES:
:CUSTOM_ID: adr-002
:END:
CLOSED: [2024-04-02 Tue]

*Status:* Accepted.
*Decision:* Default new clients to RESP3; maintain RESP2 compatibility.
*Rationale:* RESP3 supports typed responses (maps, sets) natively, reducing
client-side parsing overhead. Existing Redis clients continue to work unchanged.

*** ADR-003: Tokio over async-std as the async runtime
:PROPERTIES:
:CUSTOM_ID: adr-003
:END:
CLOSED: [2024-04-10 Wed]

*Status:* Accepted.
*Decision:* Use Tokio as the sole async runtime.
*Rationale:* Tokio has the largest ecosystem (tower, hyper, axum, tonic).
The work-stealing scheduler fits our use case better than async-std's
single-threaded per-thread-per-core model.

*** ADR-004: LWW conflict resolution using Hybrid Logical Clocks
:PROPERTIES:
:CUSTOM_ID: adr-004
:END:
CLOSED: [2024-05-20 Mon]

*Status:* Accepted.
*Decision:* Use HLC timestamps for LWW conflict resolution instead of wall-clock time.
*Rationale:* Wall-clock LWW suffers from clock skew across nodes. HLC guarantees
that =HLC(a) < HLC(b)= whenever event =a= causally precedes event =b=, even with
modest clock drift (< 500 ms). See [cite:@kulkarni2014hlc].

*** ADR-005: Pluggable eviction via trait objects
:PROPERTIES:
:CUSTOM_ID: adr-005
:END:
CLOSED: [2024-07-01 Mon]

*Status:* Accepted.
*Decision:* Model eviction policies as ~dyn EvictionPolicy~ trait objects rather
than an enum.
*Rationale:* Allows third-party eviction policies without modifying core code.
Performance cost (one virtual dispatch per eviction candidate evaluation) is
acceptable at < 1 ns overhead per call.
